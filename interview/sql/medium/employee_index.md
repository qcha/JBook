# Индексы на employee

## Условие

Пусть дана таблица в PostgreSQL:

```sql
create table employee (
    id serial primary key,
    name text,
    salary bigint,
    age int,
    sex text,
    created_at timestamp default now()
);
```

В таблице уже есть данные: примерно поровну женщин и мужчин, возраста от 0 до 70 и сгенерированы случайным образом, можно считать, что сильного перекоса нет по возрастам.

Делается следующий запрос:

```sql
select
    *
from
    employee
where
    sex = 'female'
and
    salary > 150000
and
    age = 50
order by
    created_at
```

Запрос отрабатывает за 10+ секунд и это долго. Ваша задача исправить эту ситуацию.

## Решение

Понятно, что нужно добавить индекс. Разберём каждое условие запроса:

| Колонка | Тип условия | Селективность |
| --- | --- | --- |
| `age = 50` | Точное равенство | ~1/71 ≈ 1.4% |
| `sex = 'female'` | Точное равенство | ~50% |
| `salary > 150000` | Диапазон (range) | неизвестна, зависит от данных |
| `created_at` | ORDER BY | нет |

Прежде чем рассматривать варианты, нужно вспомнить два ключевых правила составных B-tree индексов в PostgreSQL:

1. Правило range: колонка с условием `>`, `<`, `between`, `like 'x%'` должна стоять последней среди фильтрующих колонок. Всё, что стоит в индексе после range-колонки, Postgres не использует для фильтрации.
2. Правило order by: чтобы Postgres мог использовать индекс для сортировки и не делать отдельный шаг filesort, колонка из `order by` должна стоять последней в индексе, и при этом перед ней не должно быть range-условия.

### `(sex, salary, age)` и любая перестановка с `sex` первым

Ведущая колонка - это точка входа в B-tree: чем она селективнее, тем меньше страниц индекса нужно прочитать.

У `sex` всего два значения (`male`/`female`), то есть около 50% строк таблицы попадут под любое из значений. Это значит, что postgres придётся пройти половину индекса, что почти равнозначно sequence scan.

Вывод: убираем `sex` из индекса совсем.

Остаются варианты без `sex`:

```sql
(age, salary)
(age, created_at)
(age, salary, created_at)
(salary, age)
(salary, created_at)
(salary, age, created_at)
```

#### `(salary, age)`, `(salary, created_at)` и `(salary, age, created_at)`

У нас `salary` участвует в запросе как range-условие. Если поставить его первым, Postgres входит в индекс по диапазону `salary > 150000` и сканирует большой непрерывный участок дерева. Колонку `age` после range Postgres уже не использует для сужения поиска, она нужна только для фильтрации считанных строк.

Вывод: сканируем большой диапазон, потом фильтруем. Это хуже, чем войти по точному `age = 50` и только потом применить range.

#### `(age, salary)`

Ведущая колонка `age` первым — точное равенство и потом идет range-условие, как сужение выборки. Если данные в таблице подобраны таким образом, что после этого выборка будет небольшая, то `created_at` условие будет через filesort и не будет дорогим.

При этом, индекс на `(age, salary, created_at)` здесь не спасет: после range-условия на `salary` Postgres не может использовать следующие колонки индекса для сортировки, поэтому `created_at` на третьей позиции бесполезен, filesort всё равно потребуется.

Вывод: если `salary > 150000` достаточно селективен и строк после фильтрации останется немного, то это хороший выбор.

#### `(age, created_at)`

Здесь важно, что Postgres может пройти индекс в порядке `created_at` и отдать строки уже отсортированными, без filesort. Но это будет работать быстро только при условии, что большинство сотрудников с `age = 50` имеют зарплату выше 150k.

#### `(salary)`

Если `salary > 150000` отсекает, большую часть строк, то Postgres прочитает из индекса дофильтрует `age = 50` и `sex = 'female'`, и это будет быстро.

Но `(age, salary)` почти всегда лучше, даже при высокой селективности salary: `(age, salary)` делает больше работы на уровне индекса и меньше - на уровне heap (когда уже фильтруем).

Единственный аргумент за `(salary)` в одну колонку - он более общий: его проще использовать и в других запросах, без фильтра по `age`. Но для конкретного нашего запроса `(age, salary)` предпочтительнее.

#### partial index

Partial index - это индекс, построенный только по части строк таблицы, через `where` в определении индекса:

```sql
create index on employee(age, salary) where sex = 'female';
create index on employee(age, created_at) where sex = 'female';
```

Это улучшение поверх любого из финальных вариантов.

- `sex = 'female'` вшит в определение индекса, Postgres автоматически использует его при запросах с `where sex = 'female'`. Включать `sex` в колонки индекса не нужно.
- Индекс содержит только 50% строк таблицы. Меньше страниц - быстрее сканирование и меньше памяти.

Но такой индекс помогает только запросам с точным условием `sex = 'female'`. Запрос без фильтра по `sex` или с `sex = 'male'` этот индекс не использует.

### Итог

- `(age, salary)` - выбор по умолчанию, если `salary > 150000` высокоселективен (большинство сотрудников зарабатывают меньше 150k).
- `(age, created_at)` - лучше, если `salary > 150000` слабоселективен (большинство зарабатывают больше 150k). Тогда выгоднее читать строки уже в нужном порядке и фильтровать salary на лету.
